# Incident Visibility

## Purpose

Incident visibility helps customers understand what is known, what changed, and what happens next.

The experience should be calm and factual. It should avoid blame, speculation, or unsupported claims.

## Incident Timeline

Incident timelines should show:

- When the issue was first detected
- When it was acknowledged
- What status changes occurred
- What customer-facing updates were shared
- When the issue was resolved or moved to follow-up

## Customer Communication

Incident updates should be written for business owners as well as technical contacts. Technical detail can be included, but the main message should remain understandable.

## Accessibility and Review

Incident content should use headings, lists, and clear timestamps. Color should never be the only indicator of severity or state.
