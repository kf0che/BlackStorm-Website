# Educational Philosophy

BlackStorm education should make customers more confident, not more dependent.

The platform should explain monitoring, alerts, reports, device status, and support workflows in plain language. Educational content should appear where it helps, not as a wall of documentation before a customer can act.

## Core Philosophy

### Explain Before Asking

When the platform asks for information, it should explain why.

### Teach in Context

Help should appear near the task the customer is performing.

### Use Plain Language

Avoid unnecessary jargon. When technical terms are required, define them.

### Respect the Customer's Time

Use short modules, summaries, and optional deeper learning paths.

### Avoid Fear-Based Education

Do not use panic-driven security language. Help customers understand operational visibility calmly.

## Trust-Building Principles

Educational content should:

- State what BlackStorm can and cannot see
- Explain monitoring limits
- Avoid unsupported compliance claims
- Use examples from small business operations
- Make next steps clear
- Link to support when a customer needs help

## Progressive Disclosure

Use three layers:

### Quick Answer

A short answer near the action.

### Learn More

A contextual panel or article for customers who want more detail.

### Guided Practice

Future interactive labs or walkthroughs for customers who want hands-on learning.

## Accessibility Goals

Educational content should support:

- Screen reader navigation
- Keyboard access
- Clear headings
- Plain text alternatives for videos and diagrams
- Captions or transcripts for tutorial videos
- Responsive layouts
