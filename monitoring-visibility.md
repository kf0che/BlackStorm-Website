# Monitoring Visibility

Monitoring visibility is the core purpose of the BlackStorm customer portal.

The portal should help customers understand device status, alert history, operational trends, and what needs attention.

## Visibility Goals

Monitoring visibility should answer:

- Which devices are online?
- Which devices need review?
- What changed recently?
- Which alerts recovered?
- Which site is affected?
- Who received notifications?
- What should the customer check next?

## Device Cards

Device cards should present a small amount of useful information:

- Device name
- Device type
- Site or location
- Status
- Last checked time
- Response time or short note
- Link to details

## Status Indicators

Status indicators should use both text and color.

Recommended status labels:

- Online
- Needs Review
- Offline
- Paused
- Unknown

Avoid using color alone to communicate device state.

## Alert Summaries

Alert summaries should provide useful context:

- Alert type
- Related device
- Related site
- Start time
- Recovery time, if applicable
- Current state

## Customer Trust Considerations

Monitoring visibility should not imply perfect detection or guaranteed uptime.

Customers should understand that monitoring shows what BlackStorm can observe through configured checks. It is a visibility tool, not a promise that issues can never happen.

## Mobile and Responsive Considerations

Monitoring data should work on small screens:

- Device cards should stack cleanly
- Tables should scroll horizontally when needed
- Critical status labels should stay visible
- Actions should remain reachable by keyboard and touch
- Charts should have text summaries for small screens

## Future Scaling Goals

Monitoring visibility should support future:

- Multi-site customers
- MSP customer switching
- RBAC-limited views
- Alert notification systems
- Dashboard widgets
- Educational onboarding content
